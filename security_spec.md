# Field Trip Security Specification (Fortress Protocol)

Project ID: `field-trip-495823`
Status: Hardened Beta

## 1. Data Invariants
- A submission (`entry`) must belong to an existing challenge.
- A user cannot award themselves points or accolades (`stamps`).
- Invite codes (`accessCodes`) are required for registration and have usage limits.
- Users can only interact with gameplay if their `accessStatus` is `approved`.
- Administrative fields (`role`, `points`, `score`) are immutable for non-admin users.

## 2. The "Dirty Dozen" (Target Payloads)
The following payloads MUST be rejected by Firestore Security Rules:

1. **Identity Spoofing**: Attempt to create a profile for `victim_uid` signed in as `attacker_uid`.
2. **Privilege Escalation**: Update user profile to set `role: 'admin'`.
3. **Point Injection**: Update user profile to increment `points` directly.
4. **Invite Scraping**: Attempting to list all documents in `accessCodes`.
5. **Invite Poisoning**: Update an `accessCode` to reset its `currentUses`.
6. **Ghost Submission**: Creating an `entry` with a future `createdAt` timestamp.
7. **Self-Voting**: Creator of an entry casting a `vote` for their own `entryId`.
8. **Double Voting**: Casting two votes for the same entry in the same category/week.
9. **Outcome Tampering**: Updating an `activeSabotage` to set `status: 'survived'` without admin/server intervention.
10. **Bypassing Access**: Attempting to read `challenges` or `crews` when `accessStatus` is `pending`.
11. **Shadow Update**: Updating a submission to change the `userId` of the post.
12. **System Injection**: Injecting a 1MB string into the `usernames` ID field.
